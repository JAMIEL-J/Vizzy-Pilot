SR

Siddharth R.

Security Architect · OWASP

Critical — 4 exploitables

SQLite path traversal gives any logged-in user full credential read on vizzy.db. Broken access control on every analytics endpoint means user A reads user B's dashboard without owning the dataset. No column sanitization means a CSV with revenue"; DROP TABLE-- as a header reaches DuckDB unmodified. Hardcoded JWT secret means any token signed offline is valid. None of these are theoretical. All four are exploitable today in one HTTP request each.

"This is not 'security debt'. This is undeployed. The difference matters."

AM

Anita M.

Principal Backend Architect

Not horizontally scalable

Stateful file-system design fails the moment you run two containers — neither can find the other's uploads. DuckDB writer locks are a hard ceiling on concurrent ingestion. Pandas loading full CSVs into memory is an OOM bomb above 2GB. The in-memory rate limiter is per-node, not shared, so a multi-node deployment has no brute-force protection at all. The 5,300-line chart_recommender.py is operationally untestable — you cannot write a meaningful unit test for a 244KB file.

"The architecture is correct in concept but fragile in every execution detail."

PK

Priya K.

Senior Product Manager · B2B SaaS

Onboarding cliff kills retention

The Semantic Audit gate eliminates value delivery before users see any output. In B2B SaaS terms: your time-to-value is infinite because users hit a form before they see a chart. Free alternatives (Looker Studio, Tableau Public) render something immediately on upload. The "Why" driver text is Vizzy's only defensible differentiator. Without it, Vizzy is a worse version of tools that cost nothing and require no setup.

"Time-to-value above 60 seconds with no output is a churn guarantee in any SaaS product."

LF

Leo F.

Senior UX Designer · Data Products

Structural UX debt

SSE + skeleton loaders are genuinely strong — perceived performance is the best part of the current UX. The Zustand state model is the root cause of the All Columns duplication bug — state is populated by render order, not by data completeness. That class of bug reappears on every new tab addition. The filter re-aggregation patch was correct but it was a patch, not a fix. The underlying coupling between render sequence and data-layer state has not been addressed.

"Skeleton loaders are a latency mask. They don't fix a missing dashboard."

DV

Dev V.

Staff Data Engineer · Pipelines

Pipeline integrity broken

Every downstream SQL builder — DuckDB, pandas_pipeline, kpi_engine — receives raw unsanitized column names from CSV headers. Production data regularly contains spaces, slashes, parentheses, quotes, Unicode, and leading digits. Each of these breaks a different part of the pipeline in a different way: some silently corrupt, some throw unhandled exceptions, some open injection vectors. No indices on dynamic dataset tables means query latency degrades linearly with row count growth. These are data correctness issues, not just performance issues.

"Column sanitization is the first function you write when reading external data into any query engine. It is not optional."

NK

Nandini K. new

Business Analyst · Enterprise BI Rollouts

Unusable by the actual target user

I represent the user Vizzy is supposedly built for: a non-technical business stakeholder who needs answers from data without depending on a BI team. From that perspective, the current Vizzy fails at step one. Uploading a CSV and being asked to map columns to semantic types is a task I cannot perform without help — I don't know what "metric", "dimension", or "date column" means in a technical context. The charts that eventually appear have no context about what they mean or why the numbers moved. Without the "Why" driver annotation, every chart is a description of what happened with zero explanation of why, which is the only question I actually care about. Vizzy needs to feel like asking a question to a person, not configuring a data tool.

"If I need to read documentation to upload a CSV, I will go back to asking the analyst to do it for me."

RS

Rahul S. new

Senior Data Analyst · Revenue Analytics

Chart output is shallow for actual analysts

From an analyst's perspective: the auto-dashboard concept has genuine value for first-pass exploration — it saves the 20 minutes of boilerplate chart setup I'd do in Excel or Tableau. But the output stops exactly where analysis should start. The KPI detection correctly identifies what a metric is, but gives me no segmentation capability, no cohort analysis, no custom calculated fields, and no ad-hoc SQL editor. The NL2SQL chat panel is the right instinct, but if I can't write or edit the generated SQL directly, I can't trust or verify the output. Analysts don't outsource SQL to a black box — we spot-check it. The missing multi-table join capability means Vizzy is useless for any real-world dataset that isn't perfectly denormalized in a single flat file. That's almost no production dataset I've ever worked with.

"Give me the generated SQL. Let me edit it. That one change makes this a tool I'd actually open."

Riya J.

Portfolio anchor — strong with correct framing

The concept density here is exceptional for a fresher project: hybrid query engine, SSE streaming, NL2SQL intent routing, JWT+RBAC, approval-gated cleaning with immutable versioning. "The flaws are known. The fixes are documented. That combination is more credible than a clean-looking project with hidden problems."

Analyst-specific gaps — what Nandini and Rahul found

NK

Business analyst gap map

Nandini K. — what a non-technical user actually needs

Missing — blocks adoption

Zero-input first render

Upload CSV → see charts. No form, no mapping, no configuration. Business users stop at the first unfamiliar technical term.

"Why did this happen?" text

Every KPI needs a 1-sentence driver annotation. "Revenue dropped 12% — driven by churn in the Enterprise segment." Without this, the dashboard answers nothing a business user cares about.

Natural language question entry

The chat panel exists but is buried. Business users need a prominent "Ask a question about your data" input as the primary interaction, not a side panel.

Exists but needs improvement

Chart labels and context

Charts render with technical axis labels from column names. Need human-readable labels derived from the schema audit mappings.

Error messaging

Current fallback errors expose technical terms. Business users need plain-language guidance: "We couldn't detect a date column — can you tell us which one it is?"

Data cleaning approval flow

The approval-gated cleaning is a good pattern but the confirmation UI uses technical diff language. Needs a plain-language summary of what will change.

RS

Data analyst gap map

Rahul S. — what a working analyst actually needs

Missing — blocks analyst use

Editable SQL output in chat panel

NL2SQL generates SQL but doesn't show it. Analysts verify and edit SQL — a black-box query engine is not trusted in any production analytics workflow. Show the SQL, allow edits, re-run on change.

Multi-table join capability

Every real-world dataset is normalized across multiple tables. Orders, Customers, Products — three separate files. Without joins, Vizzy only works on pre-processed flat exports, which is not how production data lives.

Custom calculated fields

No way to define revenue - cost as a new column, create cohort windows, or compute rolling averages. Without custom fields, advanced metric construction requires pre-processing outside Vizzy.

Export to CSV / clipboard

Analysts cross-reference Vizzy output with other tools. No export means Vizzy is a dead-end — outputs can't leave the application.

Existing strengths to build on

SSE streaming + skeleton loaders

Fast perceived performance makes exploration feel responsive. Analysts won't tolerate full-page loads between queries.

DuckDB as the query engine

Correct choice. DuckDB is fast, SQL-compliant, handles window functions, CTEs, and lateral joins. The engine can support all the missing analyst features — it's the layer above that's missing them.

Immutable versioning + audit log

Append-only audit log with diff modal is production-grade thinking. Analysts need to trace which version of a dataset produced which chart.

Filter re-aggregation on subsets

Dynamic re-aggregation on filtered data subsets is how analysts explore — filter to a segment, see all metrics recalculate. This is a non-trivial feature done correctly.

6 phases — ordered by severity and dependency chain

1
Security hardening — 5 blockers
~3–5 days · must complete before any public deployment

Kill the SQLite path traversal

Remove SQLite from allowed connector types in external_db.py entirely. If SQLite support is required, validate that the resolved absolute path is strictly within the user's own upload directory — any traversal to a parent directory returns HTTP 400 before a connection is attempted.

Critical · external_db.py

Enforce dataset ownership on every analytics endpoint

Add a verify_dataset_owner(dataset_id, current_user) FastAPI dependency injected into every route in analytics_routes.py, chat_routes.py, and cleaning_plan_routes.py. Query dataset.owner_id and raise HTTP 403 if it doesn't match JWT subject. No exceptions, including for admin roles — those use an explicit bypass flag.

Critical · analytics_routes.py · chat_routes.py · cleaning_plan_routes.py

Rotate JWT secret, fail-fast on missing or default value

Remove the change-me-in-production default from config.py. On startup, check if SECRET_KEY env var is missing, empty, or matches a known weak value — if so, raise a startup exception that prevents the app from booting. Add a pre-commit hook that blocks any commit containing that string.

Critical · config.py · CI/CD hooks

Add rate limiting to auth endpoints + Redis-backed store

Apply slowapi limiter at 10 req/min per IP to /register, /login, /login/user, /login/admin. Move the rate limit store from in-memory dict to Redis (or Upstash for serverless) so limits are shared across all nodes. Add account lockout after 10 consecutive failures with a 15-minute cooldown.

High · auth_routes.py · Redis

Add security response headers middleware

Add FastAPI middleware emitting on every response: Content-Security-Policy (restrict to own origin), Strict-Transport-Security: max-age=63072000, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy: same-origin.

High · main.py middleware
2
Data pipeline integrity
~2–3 days · before any new ingestion format

Centralized column_sanitize() at ingestion entry point

Write a single sanitize_column_name(raw: str) → str: strip whitespace, replace all non-alphanumeric characters except underscore with underscore, collapse consecutive underscores, lowercase, prefix col_ if result starts with a digit. Call this on every column name immediately after CSV/Excel parse. Store both raw display name and safe internal name in the schema registry.

Critical · ingestion entry · coercion.py · pandas_pipeline.py

Add indices at dynamic table creation time

In the DuckDB table creation path, immediately after table build, create a composite index on detected date columns and the top 3 categorical columns flagged by the schema detector. Use CREATE INDEX IF NOT EXISTS. For date-trend queries, add a covering index on date + metric column pairs that appear in KPI engine outputs.

Medium · execution_router.py

Replace all f-string SQL interpolation with parameterized queries

Audit every f"... {column} ..." pattern in coercion.py and pandas_pipeline.py. Replace with DuckDB's parameterized API or SQLGlot AST column builders. Sanitization at ingestion plus parameterized execution is defense-in-depth — neither alone is sufficient.

High · coercion.py · pandas_pipeline.py
3
Product UX — remove the onboarding gate
~5–7 days frontend · before any user acquisition

Auto-render dashboard using LLM mappings as defaults

Change ingestion flow: upload → LLM mapping → dashboard renders immediately with a dismissable "Review mappings" banner. Low-confidence columns (<0.6 score) render with a yellow badge, still visible. Semantic Audit becomes an optional edit action triggered from the banner or individual chart labels — never a gate.

Critical UX · ingestion flow · frontend

Add "Why" driver text annotations to all KPI charts

Run Pearson/Spearman correlation on the pre-aggregated ~50-row dataset after dashboard generation. For each KPI, find top 1–2 correlated dimensions. Generate: "[Metric] moved X% — driven by [Dimension] segment [Value]." Render below each chart card. This is the single feature separating Vizzy from free static chart tools and is the only thing Nandini K. said she needed to recommend it to her team.

High value · kpi_engine.py · frontend

Promote NL question input to primary interaction pattern

Move the natural language query input from a side panel to a prominent position above the dashboard — visually the first thing users see. "Ask anything about your data" placeholder. This is the interaction model business users already understand from ChatGPT. The side panel can persist for deeper sessions but the primary entry point needs to be unavoidable.

High UX · frontend · UserDashboard.tsx
4
Analyst capability — SQL transparency + joins
~1–2 weeks · required for analyst user adoption

Expose generated SQL in chat panel with inline editor

After NL2SQL generates a query, display the SQL in a collapsible code block below the chart output. Add an "Edit SQL" button that opens the SQL in an inline editor (CodeMirror or Monaco). Re-run on save. This is the single change Rahul S. said would make Vizzy a tool he'd actually open in a production workflow. The SSE streaming already handles re-execution — this is a frontend exposure change, not a backend redesign.

High analyst value · chat panel frontend · SSE execution

Multi-table upload + visual join builder

Allow uploading multiple CSVs in one session. DuckDB already registers each as a separate table — expose this in the UI as a schema sidebar. Add a drag-to-join builder: user drags a column from Table A onto a column from Table B to define a join key. On confirm, the joined view becomes the active dataset for all dashboard charts. This is the highest-impact single feature for analyst adoption.

High · multi-upload · join builder · DuckDB view creation

Export to CSV and clipboard for every chart result set

Add a download icon to every chart card that exports the underlying aggregated data as a CSV. Add a "Copy" option that puts tab-separated values in clipboard for direct paste into Excel or Sheets. The underlying data is already in DuckDB — this is a serialization-to-response operation, not a new query.

Medium · frontend chart actions · DuckDB fetch

~1–2 weeks · before adding any new analytics feature

Split chart_recommender.py into 5 domain modules

recommender/base.py (routing), key_insights.py (Phase A/B), all_columns.py (Phase C/D), domain_detector.py (KPI classification), chart_types.py (config factories). Each under 400 lines. Public interface unchanged — callers import from recommender package.

DashboardLayout, ChartGrid, FilterPanel, SemanticOverlay (now optional), ChatPanel, CleaningModal. Each owns its Zustand slice. No shared mutable state between siblings. Decouple chart config population from render order — use a dataVersion field for invalidation.

Tech debt · UserDashboard.tsx

Unit tests for execution_router and kpi_engine critical path

Write pytest tests covering: DuckDB vs Pandas routing threshold, filter re-aggregation correctness, SSE completion signaling, column fallback behavior. These tests also validate the 104ms p95 claim used in portfolio framing — without them it's an unverified number.

Medium · execution_router.py · kpi_engine.py
6
Scalability — production infrastructure
~2–4 weeks · only required if scaling beyond demo

S3/R2 object storage + MotherDuck analytical layer

Abstract storage behind a DatasetStorageAdapter interface (local and cloud implementations). Upload raw CSVs to S3 or Cloudflare R2 via presigned URLs. Replace local DuckDB file builds with MotherDuck cloud connections. API server becomes fully stateless — horizontal scaling works.

Scalability · ingestion · execution_router.py

Background task queue for ingestion (Celery + Redis or Temporal)

Upload endpoint returns job_id immediately. CSV → DuckDB conversion runs in background worker. Frontend subscribes to job-status SSE stream. Decouples ingestion latency from HTTP lifecycle. Prevents worker OOM from killing the API server.

Scalability · background jobs

Add structured logging + observability layer

Add structured JSON logging via structlog. Instrument execution_router.py with query duration metrics (this validates your 104ms p95 claim with real data). Add Sentry for error tracking. Add uptime monitoring via BetterStack or equivalent. Without this, you're flying blind in production.

Council's final statement

Vizzy is not production-ready today — not even close. Five security issues are exploitable right now by any authenticated user, the onboarding flow eliminates value delivery before most users see a chart, and the two new analyst council members confirm that the current output depth is too shallow for either of the two user personas the product claims to serve. That's the honest assessment.

The architectural instincts underneath are genuinely correct: hybrid DuckDB/Pandas routing, SSE streaming, LLM-constrained to schema inference only, approval-gated cleaning with immutable versioning, JWT+RBAC — none of these are accidental choices. They're the right choices. The problem is that the security layer was never hardened, the UX never went past MVP, and the analyst capabilities were never built past the first-pass auto-dashboard concept.

The remediation path is clear and the phases are ordered. Phase 1 and 2 — security and pipeline integrity — are non-negotiable prerequisites for sharing the URL publicly. They take about a week. Phase 3 — removing the onboarding gate and adding "Why" driver text — is what makes Vizzy worth showing to a business user. Phase 4 — SQL transparency and multi-table joins — is what makes it worth showing to an analyst. After all four phases, Vizzy is a production-viable tool for a specific, well-defined use case: zero-setup exploratory analytics on single or joined CSV datasets, with NL-first query interaction and automated causal annotations.

That is a real, defensible product. It is not Tableau. It is not Snowflake. It is not trying to be. And that specificity — combined with honest framing of what it is and what it isn't — is exactly what makes it a credible portfolio anchor.
